# Research: MongoDB vs PostgreSQL (UGC workload)

## Краткое резюме

В рамках исследования было сгенерировано и протестировано более **10 млн записей UGC** (лайки, закладки, рецензии).
PostgreSQL показал **существенно меньшую задержку на чтение** и более быстрые batch-вставки.
MongoDB продемонстрировал **предсказуемую производительность** и хорошее поведение при realtime write+read.
Обе СУБД укладываются в требования по latency, но имеют разные trade-off’ы.
Результаты подтверждают, что выбор хранилища должен зависеть от характера нагрузки.

---

## Цель исследования

Сравнить MongoDB и PostgreSQL как хранилища для пользовательского контента (UGC):

* лайки
* закладки
* рецензии

Критерии сравнения:

* скорость генерации данных (bulk write)
* latency на чтение типовых запросов
* latency на realtime write+read
* поведение на объёмах **10+ млн записей**

---

## Нагрузка и модель данных

### Общие параметры

* Пользователи: **50 000**
* Фильмы: **10 000**
* Лайки: **7 500 000**
* Закладки: **3 000 000**
* Рецензии: **200 000**
* Общий объём: **~10.7 млн записей**

Типовые операции:

* список лайков пользователя
* количество лайков/дизлайков фильма
* средний рейтинг фильма
* список закладок пользователя
* realtime сценарий: запись + немедленное чтение

---

## MongoDB

### Генерация данных

```
LIKES:     ~7 500 000   → 807.64s
BOOKMARKS: ~3 000 000   → 287.49s
REVIEWS:      200 000   → 15.34s
```

### Чтение (ms)

```
list_user_likes          avg=2.53  p95=3.75
film_like_dislike_cnt   avg=2.20  p95=3.42
film_avg_rating         avg=1.53  p95=2.49
list_user_bookmarks     avg=1.45  p95=2.45
```

### Realtime write + read (ms)

```
avg=5.21  p95=7.15
```

### Вывод по MongoDB

* Предсказуемая latency даже на больших объёмах
* Хорошо подходит для realtime сценариев
* Bulk-вставки заметно медленнее PostgreSQL

---

## PostgreSQL

### Генерация данных

```
LIKES:     ~7 500 000   → 102.69s
BOOKMARKS: ~3 000 000   → 20.00s
REVIEWS:      200 000   → 12.60s
```

### Чтение (ms)

```
list_user_likes          avg=0.63  p95=0.82
film_like_dislike_cnt   avg=1.11  p95=1.49
film_avg_rating         avg=0.74  p95=1.08
list_user_bookmarks     avg=0.59  p95=0.84
```

### Realtime write + read (ms)

```
avg=3.85  p95=5.06
```

### Вывод по PostgreSQL

* Существенно быстрее на чтении
* Очень быстрые batch-вставки
* Отлично подходит для аналитических и read-heavy сценариев

---

## Сравнительная таблица

| Метрика                 | MongoDB    | PostgreSQL |
| ----------------------- | ---------- | ---------- |
| Bulk insert             | ❌ медленно | ✅ быстро   |
| Read latency            | ⚠️ выше    | ✅ ниже     |
| Realtime write+read     | ⚠️         | ✅          |
| Предсказуемость latency | ✅          | ✅          |
| Простота схемы          | ✅          | ⚠️         |

---

## Итоговый вывод

* **PostgreSQL** предпочтителен для read-heavy UGC и аналитических запросов.
* **MongoDB** подходит для гибких схем и realtime сценариев с умеренной нагрузкой.
* В реальных системах возможен гибридный подход (OLTP + аналитика).

Данное исследование подтверждает, что оба решения масштабируются до **10+ млн записей** с приемлемой latency.
