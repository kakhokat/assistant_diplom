# Сервис авторизации онлайн-кинотеатра

Сервис авторизации обеспечивает:

- регистрацию и аутентификацию пользователей;
- выдачу и обновление JWT-токенов (access + refresh);
- управление ролями (RBAC) и проверку прав доступа;
- вспомогательные функции личного кабинета (изменение логина/пароля, история входов\*).

\* — функции со звёздочкой можно опустить при индивидуальном выполнении работы, как допускает задание.

---

## 1. Цели и требования

### Функциональные требования

- Регистрация пользователя (`/api/v1/auth/signup`).
- Вход в систему с обменом логина/пароля на пару токенов (`/api/v1/auth/login`).
- Обновление access-токена по refresh-токену (`/api/v1/auth/refresh`).
- Выход из текущего аккаунта (`/api/v1/auth/logout`).
- Выход из остальных аккаунтов / устройств без хранения всех access-токенов в БД (`/api/v1/auth/logout-others`).
- Получение профиля текущего пользователя и его ролей (`/api/v1/auth/me`).
- Управление личным кабинетом:
  - изменение логина / пароля (`/api/v1/users/me`) \*;
  - получение истории входов (`/api/v1/users/me/login-history`) \*.
- Управление ролями:
  - CRUD ролей (`/api/v1/roles`);
  - назначение и отзыв ролей (`/api/v1/roles/assign`, `/api/v1/roles/revoke`).
- Проверка прав доступа для других сервисов (`/api/v1/permissions/check`).

### Нефункциональные требования

- REST API, описанное через OpenAPI (см. `openapi.yaml`).
- Асинхронный фреймворк **FastAPI**.
- Хранение пользователей и ролей в **PostgreSQL**.
- Хранение недействительных access-токенов в **Redis** (чёрный список).
- Возможность горизонтального масштабирования (все данные для проверки токенов лежат в БД/Redis, а не в памяти процесса).
- Код соответствует PEP8, форматируется `black`/`isort`, проверяется `flake8`.
- Ключевые ручки и бизнес-логика покрыты тестами.

## 2. Архитектура системы

### 2.1. Общая схема

Сервис авторизации — отдельный микросервис в инфраструктуре онлайн-кинотеатра.

Основные компоненты:

- **FastAPI-приложение** — HTTP-интерфейс.
- **PostgreSQL** — основное хранилище пользователей, ролей, refresh-токенов и истории входов.
- **Redis** — кэш и чёрный список access-токенов.
- **Nginx** (опционально) — фронт-прокси перед API.
- **Gunicorn + Uvicorn** — продакшн-сервер для FastAPI.

Типичная схема запросов:

`Клиент → Nginx → Auth Service (FastAPI) → PostgreSQL / Redis`

### 2.2. Архитектурные слои

Проект разделён на несколько слоёв с чёткими ответственностями:

- `api/` — HTTP-ручки, входной слой (FastAPI routers). Содержит только:
  - декларацию маршрутов;
  - привязку схем запрос/ответов;
  - обработку HTTP-специфичных деталей (status codes, headers).
  - ВСЯ бизнес-логика вынесена в `services/`.

- `services/` — слой бизнес-логики:
  - регистрация и вход пользователей;
  - выпуск и валидация токенов;
  - управление ролями и проверка прав;
  - работа с историей входов и токенами.
  - Не знает про HTTP напрямую — работает через абстракции репозиториев.

- `domain/` — доменные модели и порты:
  - `domain/models` — Pydantic-модели (DTO), описывающие данные на границах слоёв.
  - `domain/ports` — протоколы репозиториев и кэша (`UserRepository`, `RoleRepository`, `TokenRepository`, `Cache`).

- `infrastructure/` — адаптеры для конкретных технологий:
  - `infrastructure/postgres` — реализации репозиториев на SQLAlchemy (PostgreSQL).
  - `infrastructure/redis` — реализация кэша и чёрного списка токенов.

- `db/` — создание и конфигурация подключений:
  - `db/postgres.py` — `AsyncEngine`, `AsyncSession`, базовый класс `Base`.
  - `db/redis.py` — клиент Redis и DI-функции для FastAPI.

- `core/` — общие настройки и инфраструктурные компоненты:
  - `core/settings.py` — конфигурация приложения через Pydantic Settings.
  - `core/security.py` — JWT, работа с паролями (hash/verify), создание `jti`, `token_version`.
  - `core/logger.py` — настройки логгера.

## 3. Модель доступа (RBAC)

Используется **Role-Based Access Control (RBAC)**:

- Пользователь может иметь несколько ролей.
- Доступ к операциям определяется наличием роли/ролей.
- В JWT содержатся:
  - `sub` — ID пользователя;
  - `roles` — список ролей пользователя;
  - `is_superuser` — флаг суперпользователя;
  - `token_version` — версия токенов пользователя (для logout-others);
  - `jti` — уникальный идентификатор токена.

Минимальный набор ролей:

- `user` — обычный зарегистрированный пользователь.
- `subscribers` — подписчик, может смотреть «фильмы младше 3 лет».
- `admin` — управление пользователями и ролями.
- `superuser` — полный доступ, обходящий проверки ролей.

Использование в онлайн-кинотеатре:

- Контент-сервис, получая JWT, проверяет:
  - что токен валиден и не просрочен;
  - что у пользователя есть роль `subscribers` для просмотра нового контента;
  - либо вызывает `/api/v1/permissions/check` auth-сервиса.

## 4. Хранилища и схема данных

### 4.1. PostgreSQL

Основные таблицы:

- `users`
- `roles`
- `user_roles` (связь многие-ко-многим)
- `refresh_tokens`
- `login_history`

Краткое описание полей:

**users**

- `id: uuid` — PK;
- `login: varchar(255)` — уникальный логин;
- `password_hash: varchar(255)` — захэшированный пароль;
- `is_active: bool` — активен ли пользователь;
- `is_superuser: bool` — суперпользователь;
- `token_version: int` — версия токенов (используется для Logout Others);
- `created_at: timestamptz` — дата создания пользователя.

**roles**

- `id: uuid` — PK;
- `name: varchar(50)` — уникальное имя роли;
- `description: text` — описание роли.

**user_roles**

- `id: uuid` — PK;
- `user_id: uuid` → `users.id` (ON DELETE CASCADE);
- `role_id: uuid` → `roles.id` (ON DELETE CASCADE);
- `UNIQUE(user_id, role_id)` — одна и та же роль не может быть назначена пользователю дважды.

**refresh_tokens**

- `id: uuid` — PK;
- `user_id: uuid` → `users.id`;
- `token_hash: varchar(255)` — хэш refresh-токена;
- `user_agent: varchar(255)` — информация об устройстве;
- `client_ip: inet / varchar` — IP;
- `created_at: timestamptz`;
- `expires_at: timestamptz`;
- `revoked: bool` — отозван ли токен.

**login_history**

- `id: uuid` — PK;
- `user_id: uuid` → `users.id`;
- `login_at: timestamptz` — время события;
- `user_agent: varchar(255)`;
- `client_ip: inet / varchar`;
- `action: varchar(50)` — тип события (`login`, `logout`, `refresh`, `logout_all`).

### 4.2. Redis

Используется для:

1. **Чёрный список access-токенов**:

   - Ключ: `blacklist:<jti>`.
   - Значение: произвольная строка (например, `1`).
   - TTL: время до истечения `exp` токена.

2. Дополнительные временные данные (при необходимости).

## 5. JWT и сессии

### 5.1. Access-токен

- Срок жизни: по умолчанию 10–15 минут (настраивается через конфиг).
- Не хранится в БД.
- Payload включает:
  - `sub` — UUID пользователя;
  - `roles` — список строковых ролей;
  - `is_superuser: bool`;
  - `token_version: int`;
  - `jti: str` — уникальный ID токена;
  - `exp`, `iat`, `nbf`.

Проверка access-токена:

1. Валидация подписи и времени жизни.
2. Проверка, что `jti` отсутствует в чёрном списке Redis.
3. Сверка `token_version` из токена с текущим значением в таблице `users`.

### 5.2. Refresh-токен

- Долгоживущий токен (например, 30 дней).
- Всегда хранится в БД в виде **хэша**, а не «сырого» токена.
- При каждом `refresh` токен может:
  - либо «переиспользоваться» до истечения срока (+ проверка revoked),
  - либо ротироваться (выдаётся новый, старый помечается `revoked=true`).

## 6. Logout и «Выйти из остальных аккаунтов»

### 6.1. Logout (текущая сессия)

- Текущий refresh-токен помечается `revoked=true`.
- Текущий access-токен добавляется в чёрный список Redis с TTL до `exp`.
- В `login_history` создаётся запись с `action='logout'`.

### 6.2. Logout Others (дополнительное задание)

- Увеличивается `users.token_version` на 1.
- Все refresh-токены пользователя помечаются `revoked=true` (кроме того, который используется в текущей сессии, если нужно его сохранить).
- Все **старые access-токены** немедленно становятся невалидными, т.к. в них записано старое значение `token_version`.

Важно: при этом **не требуется** хранить все активные access-токены в БД.

## 7. API и обработка ошибок

### 7.1. Основные эндпоинты

Кратко (подробно см. `openapi.yaml`):

- `POST /api/v1/auth/signup` — регистрация.
- `POST /api/v1/auth/login` — вход, выдача `access` + `refresh`.
- `POST /api/v1/auth/refresh` — обновление токенов.
- `POST /api/v1/auth/logout` — выход из текущего аккаунта.
- `POST /api/v1/auth/logout-others` — выход из остальных аккаунтов.
- `GET /api/v1/auth/me` — профиль текущего пользователя.
- `PATCH /api/v1/users/me` — изменение логина и/или пароля\*.
- `GET /api/v1/users/me/login-history` — история входов\*.
- `GET /api/v1/roles` — список ролей.
- `POST /api/v1/roles` — создать роль.
- `PATCH /api/v1/roles/{role_id}` — изменить роль.
- `DELETE /api/v1/roles/{role_id}` — удалить роль.
- `POST /api/v1/roles/assign` — назначить роль пользователю.
- `POST /api/v1/roles/revoke` — отобрать роль у пользователя.
- `POST /api/v1/permissions/check` — проверить наличие прав у пользователя.

### 7.2. Коды ответов и ошибки

Используются стандартные HTTP-коды:

- `200 OK` / `201 Created` — успешные операции.
- `204 No Content` — успешные операции без тела (logout, assign/revoke).
- `400 Bad Request` — ошибка валидации или некорректные данные запроса.
- `401 Unauthorized`:
  - отсутствует токен;
  - токен истёк;
  - токен не прошёл подпись;
  - токен в чёрном списке;
  - refresh-токен истёк/отозван.
- `403 Forbidden` — недостаточно прав (нет нужной роли, не admin).
- `404 Not Found` — пользователь, роль или refresh-токен не найдены.
- `409 Conflict` — логин занят, роль уже назначена / отсутствует.
